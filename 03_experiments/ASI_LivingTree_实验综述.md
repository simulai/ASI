## H26: Living Tree 基础

### 目标
验证节点隔离机制 + O(1)增量更新

### 核心机制
- **节点隔离**：每个任务存储为独立节点，不更新已有节点
- **O(1)增量**：只添加新节点，不修改旧节点
- **最近邻匹配**：预测时找最相似节点

### 实验设计
- 12个Python任务（loop/recursion/conditional/comprehension × 4种DO）
- 顺序学习，回测旧任务准确率
- MLP baseline对比

### 实验结果

| 指标 | Living Tree | MLP | 结论 |
|------|-------------|-----|------|
| 最终旧任务保留率 | 97.1% | 54.3% | ✓ LT无遗忘 |
| 时间增长 | O(1) | O(n) | ✓ LT常数时间 |

### 核心发现
- ✓ 节点隔离有效防止灾难性遗忘
- ✓ O(1)增量更新时间不随任务数增长
- ✗ MLP出现明显遗忘（54.3%保留）

### 下一步
- 加入CF投影（语义理解）
- 加入DO分类器（语法识别）

## H33a: CF投影器

### 目标
用嵌入投影实现控制流（CF）分类

### 方法
- 训练投影矩阵W_cf将768维嵌入投影到128维CF空间
- 三元组损失：同类接近，异类远离
- 最近原型分类

### 实验设计
- 4类CF：loop, recursion, conditional, comprehension
- 120个训练样本，60个测试样本

### 实验结果

| 指标 | 结果 | 目标 |
|------|------|------|
| CF分类准确率 | 60.9% | 70%+ |
| 混淆主要在 | conditional↔comprehension | 语义重叠 |

### 核心发现
- ✓ CF投影可以区分loop/recursion
- △ conditional/comprehension易混淆（语义相似）
- △ 小样本下准确率受限

### 下一步
- 增大训练集
- 加入DO语法增强

## H33b: DO语法分类器

### 目标
用关键词匹配实现数据对象（DO）分类

### 方法
- 维护DO关键词表：list/dict/set/arithmetic
- 统计代码中关键词出现频率
- 归一化后取最大值

### 实验设计
- 4类DO：list, dict, set, arithmetic
- 80个样本测试

### 实验结果

| 指标 | 结果 | 目标 |
|------|------|------|
| DO分类准确率 | 73.9% | 90%+ |
| 混淆主要在 | 混合类型代码 | 边界模糊 |

### 核心发现
- ✓ 关键词匹配简单有效
- △ 混合代码（如同时用list和dict）难以分类

### 下一步
- 扩展关键词表
- 加入上下文感知

## H33: Living Tree + H33a策略集成

### 目标
验证H33a策略（CF投影+DO语法）能否集成到Living Tree

### 核心机制
- **节点隔离**：每个任务独立存储
- **CF投影**：H33a训练的投影器
- **DO语法**：H33b的关键词分类器
- **O(1)增量**：只添加节点

### 实验设计
- 12个任务，48个样本
- 训练集：每个任务第1个样本（11个，用于CF训练）
- 测试：顺序学习后回测旧任务

### 实验结果

| 指标 | 结果 | 评价 |
|------|------|------|
| LT旧任务保留率 | 100% | ✓ 隔离有效 |
| MLP旧任务保留率 | 100% | △ 评估失效 |
| LT时间增长 | 0.58x | ✓ O(1) |
| CF准确率 | - | 无法评估 |
| DO准确率 | - | 无法评估 |

### 核心发现
- ✓ 节点隔离机制有效（100%保留）
- ✓ O(1)增量有效
- ✗ MLP评估方法失效（重建误差不能衡量遗忘）
- △ CF/DO评估不完整

### 下一步
- 改进MLP对比评估
- 在更大规模数据集上验证

## H35: HumanEval Full Benchmark

### 目标
在真实HumanEval benchmark上验证完整的Living Tree系统

### 实验设计
- 30个真实HumanEval任务
- 70%训练，30%测试
- 完整prompt+canonical_solution作为代码

### 实验结果

| 指标 | 结果 | 目标 | 状态 |
|------|------|------|------|
| no_forgetting | 73.7% | >75% | ✗ |
| time_constant | 0.15x | <3x | ✓ |
| cf_works | 27.3% | >30% | ✗ |
| do_works | 81.8% | >50% | ✓ |

### 核心发现
- ✓ DO准确率高（81.8%），关键词表在真实代码上有效
- ✓ 时间增长极小（0.15x），O(1)增量确认
- △ CF准确率低（27.3%），CF分布不均衡
- △ 旧任务保留低（73.7%），CF分类错误累积导致

### 分析（来自用户）
- no_forgetting 73.7%不是遗忘问题，是CF分类器在长尾分布上的局限
- cf_works 27.3%测试集只有11个样本，统计意义极低

## H36: Retrieval + Generation（Kaggle运行成功）

### 目标
从"分类系统"变为"检索+生成系统"，验证节点存储的知识是否真的有用

### 核心变化
- 不只分类CF/DO
- 用匹配节点的solution作为预测答案
- 测试节点存储的知识是否可以迁移

### 实验设计
- 20个HumanEval任务，14训练，6测试
- 给定新问题，用embedding相似度找最相似节点
- 用匹配节点的solution作为预测
- 评估exact match和key pattern match

### 实验结果

| 指标 | 结果 | 目标 | 状态 |
|------|------|------|------|
| retrieval_useful | 70.6% | >30% | ✓ |
| time_constant | 0.14x | <3x | ✓ |
| cf_retrieval | 50.0% | >25% | ✓ |
| do_retrieval | 66.7% | >40% | ✓ |

### 核心发现
- ✓ **节点存储的知识真的有用** - 检索solution评分70.56%
- ✓ **O(1)增量有效** - 时间增长0.14x
- △ **embedding相似≠CF类型相同** - 高相似度匹配但CF类型不同是常见情况
- △ **语义相似但模式不同** - 代码结构相似，但解决思路不同

### 关键洞察
1. Living Tree的检索机制可以找到语义相似的代码
2. 找到的solution在40-100%的情况下可以直接或修改后使用
3. CF/DO分类是辅助手段，不是唯一判断标准

### 下一步方向
- **方向A（修补）**：CF均衡采样，提高分类准确率
- **方向B（跨越）**：结合LLM的生成能力，用检索到的节点作为few-shot context
- **方向C（理论）**：信息分布不均匀时的自适应压缩策略

### 目标
在真实HumanEval benchmark上验证完整的Living Tree系统

### 实验设计
- 30个真实HumanEval任务
- 70%训练，30%测试
- 完整prompt+canonical_solution作为代码

### 预期结果
| 指标 | 目标 | 预期 |
|------|------|------|
| 旧任务保留率 | >75% | 90%+ |
| 时间增长 | <3x | O(1) |
| CF准确率 | >30% | 50%+ |
| DO准确率 | >50% | 70%+ |
