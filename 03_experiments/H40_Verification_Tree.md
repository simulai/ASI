## 模块定义

### 模块A：持续学习机制（已完成部分）

**验证目标**：节点隔离能否实现持续学习不遗忘

**已完成实验**：
| 实验 | 规模 | 结果 | 状态 |
|------|------|------|------|
| H26 | 小规模 | 基础机制可行 | ✅ |
| H35 | HumanEval | 73.7%保留 | ✅ |
| H36 | 检索+生成 | 70.6%评分 | ✅ |
| H38 | 100任务 | 80.2%评分 | ✅ |

**待验证**：
- [ ] A1.1：更大规模（1000+节点）
- [ ] A1.2：跨领域迁移（代码→数学）
- [ ] A1.3：长期记忆稳定性

**失败处理**：如果A失败 → 研究节点失效机制 → 改进存储结构

---

### 模块B：Hopfield动力学（核心替换）

**验证目标**：能否用能量驱动的吸引子替代QKV Attention

**关键问题**：
1. Cosine Similarity ≠ Hopfield动力学
2. 需要实现：能量函数、梯度下降到吸引子

**待验证**：
- [ ] B1：纯Hopfield检索 vs Cosine Similarity检索
- [ ] B2：多吸引子组合能力
- [ ] B3：Hopfield容量 vs Transformer上下文

**实验设计**：
```
对比实验：
1. Living Tree (Cosine) - baseline
2. Living Tree (Hopfield能量) - 核心变量
3. 随机检索 + LLM - 对照

测量：
- 检索准确率
- 组合任务准确率
- 推理延迟
```

**失败处理**：如果B失败 → 调试Hopfield参数 → B+改进或B++组合

---

### 模块C：Landauer代价感知

**验证目标**：系统能否感知"计算代价"，做出Pareto最优决策

**关键问题**：
- 什么时候"派小艇出去探测"（具身互动）？
- 什么时候靠"航海图"（内部记忆）就够了？

**待验证**：
- [ ] C1：Landauer代价项是否影响路由决策
- [ ] C2：能否学习"代价-精度"Pareto前沿
- [ ] C3：与MetaGate的交互

**实验设计**：
```
对比实验：
1. 无Landauer代价（baseline）
2. 有Landauer代价（核心变量）
3. 固定代价阈值（对照）

测量：
- 任务准确率
- 平均计算代价
- 代价-精度Pareto前沿
```

---

### 模块D：互补门控（信息隔离）

**验证目标**：信息能否通过互补门选择性隔离

**关键问题**：
- 类比：防水舱门，防止一个舱进水导致整艘船沉没
- 类比：DNA双螺旋硬配对，不是概率性软绑定

**待验证**：
- [ ] D1：互补门能否阻止错误信息传播
- [ ] D2：模块间干扰是否降低
- [ ] D3：与Hopfield吸引子的交互

---

### 模块E：自生路由（Winner-Take-All）

**验证目标**：系统能否自发生成路由决策

**关键问题**：
- 不需要softmax软分配
- 能量景观里的硬性选择
- "哪个专家的吸引子先收敛，谁就赢"

**待验证**：
- [ ] E1：自生路由 vs 显式路由
- [ ] E2：路由稳定性
- [ ] E3：多路由场景

---

## 最终验证：复杂推理基准

### 验证目标

Living Tree能否在复杂推理任务上达到或超越Transformer

### 基准选择

| 基准 | 类型 | 说明 |
|------|------|------|
| MATH | 数学推理 | 需要多步推理 |
| ARC | 抽象推理 | 需要组合泛化 |
| HumanEval | 代码生成 | 需要精确匹配 |
| GSM8K | 数学应用题 | 需要混合领域 |

### 验证标准

**"替代"定义**：
- 在同等计算资源下，准确率不低于Transformer
- 在持续学习场景下，遗忘率显著低于Transformer
- 在新任务上，组合已有知识的能力优于Transformer

## 决策规则

### 何时判定"模块验证通过"？

| 条件 | 说明 |
|------|------|
| 核心假设验证 | 统计显著高于baseline |
| 失效模式理解 | 知道为什么会失败 |
| 可改进性确认 | 知道如何修复 |

### 何时判定"研究方向失败"？

| 条件 | 说明 |
|------|------|
| 多次尝试仍失败 | 同一个问题多次实验无法解决 |
| 根本性障碍 | 理论缺陷，非工程问题 |
| 被其他方法超越 | 其他方案显著更好 |

### 何时"回到上一步"？

| 情况 | 操作 |
|------|------|
| 模块验证失败 | 改进模块或回到决策点 |
| 需要更多基础设施 | 先完成依赖模块 |
| 发现新问题 | 更新验证树 |

## 总结

这张验证树确保：

1. **系统性**：每个模块都被独立验证
2. **可追溯**：知道为什么失败，为什么成功
3. **可调整**：可以根据结果更新验证树
4. **防遗忘**：所有决策点都被记录

**核心原则**：验证驱动研究，而非假设驱动研究。
